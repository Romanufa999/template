# Skill Creator — Создание и улучшение навыков Claude Code

Создание новых навыков (skills), редактирование и улучшение существующих.
Используй этот скилл когда пользователь хочет:
1. Создать новый навык / скилл / skill с нуля
2. Отредактировать или улучшить существующий навык
3. Превратить текущий рабочий процесс в переиспользуемый навык
4. Пользователь говорит "создай скилл", "сделай навык", "turn this into a skill", "напиши скилл для..."

## Структура навыков в проекте

Навыки хранятся в `.claude/commands/*.md` — простые Markdown-файлы.
Каждый навык регистрируется в `CLAUDE.md` в таблице навыков.

## Процесс создания навыка

### Шаг 1 — Сбор требований

Выясни у пользователя:
1. **Что** навык должен делать? (основная задача)
2. **Когда** навык должен срабатывать? (триггерные фразы, контексты)
3. **Какой формат выхода?** (файлы, сообщения, API-вызовы)
4. **Есть ли зависимости?** (API-ключи, внешние сервисы, MCP-серверы)
5. **Есть ли примеры?** (реальные кейсы использования)
6. **Нужны ли подскиллы?** Если навык сложный и включает несколько независимых блоков логики — предложи разбить на главный скилл + подскиллы.

Если текущий разговор уже содержит рабочий процесс, который пользователь хочет превратить в навык — извлеки ответы из истории разговора: используемые инструменты, последовательность шагов, поправки пользователя, форматы ввода/вывода.

### Шаг 2 — Написание навыка

Структура файла `.claude/commands/<skill-name>.md`:

```markdown
# Название навыка

Краткое описание — 1-2 предложения.

Используй этот скилл когда нужно:
1. Триггер 1
2. Триггер 2
3. Триггер 3
4. Конкретные фразы пользователя

## Подскиллы

При активации этого скилла автоматически загружай следующие подскиллы:

| Подскилл | Описание | Когда применять |
|----------|----------|-----------------|
| `название-1` | Что делает | В каком контексте читать |
| `название-2` | Что делает | В каком контексте читать |

Подскиллы загружаются из таблицы `sub_skills` по `skill_id` этого скилла.

## Зависимости (если есть)

- API-ключи, переменные окружения
- Внешние сервисы

## Процедура

1. Шаг 1
2. Шаг 2
3. ...

## Формат запроса / ответа (если есть API)

Описание полей, примеры curl-запросов, JSON-схемы.

## Примеры

### Пример 1: ...
Input: ...
Output: ...

## Обработка ошибок

- Что делать если X
- Что делать если Y
```

### Принципы написания

- **Объясняй "зачем"**, а не только "что делать". Claude умный — понимание причин работает лучше жёстких правил.
- **Будь конкретным.** Указывай точные URL, форматы, имена полей. Избегай абстрактных описаний.
- **Триггеры должны быть "напористыми".** Claude склонен не активировать навыки когда стоило бы. Делай описание триггеров широким. Вместо "Используй когда нужно загрузить файл на S3" — "Используй когда нужно загрузить, задеплоить, опубликовать сайт, файлы, страницу на хостинг, S3, сервер".
- **Держи файл до 500 строк.** Если больше — вынеси детали в отдельные reference-файлы.
- **Включай примеры.** Реальные примеры использования помогают Claude правильно интерпретировать навык.
- **Избегай MUST/NEVER без причины.** Лучше объясни почему что-то важно.

### Шаг 3 — Регистрация в CLAUDE.md

Добавь строку в таблицу навыков в `CLAUDE.md`:

```markdown
| skill-name | Описание навыка | [.claude/commands/skill-name.md](.claude/commands/skill-name.md) |
```

### Шаг 4 — Тестирование

После написания навыка проверь его на 2-3 реалистичных промптах:

1. Придумай типичные запросы пользователя, которые должны активировать навык
2. Придумай пограничные случаи — запросы, похожие на тему навыка, но не совсем
3. Выполни каждый промпт, следуя инструкциям навыка
4. Покажи результаты пользователю и спроси, всё ли верно

### Шаг 5 — Итерации

На основе обратной связи:
1. Исправь проблемы в навыке
2. Перезапусти тестовые кейсы
3. Повтори до тех пор, пока пользователь не будет доволен

**Обобщай фидбек.** Навык будет использоваться много раз для разных задач. Не оптимизируй под конкретные примеры — делай инструкции универсальными.

## Редактирование существующего навыка

1. Прочитай текущий файл навыка
2. Пойми, что нужно изменить
3. Внеси изменения, сохраняя существующую структуру
4. Обнови описание в CLAUDE.md если оно изменилось
5. Проверь что навык всё ещё корректно срабатывает

## Создание подскиллов

Подскиллы — это вложенные блоки инструкций, которые автоматически загружаются при активации главного скилла. Используй подскиллы когда:
- Навык слишком большой (>300 строк) и логически делится на части
- Разные части навыка применяются в разных контекстах (один подскилл для статики, другой для SSR)
- Блок логики может переиспользоваться в нескольких скиллах

### Когда НЕ нужны подскиллы

- Навык простой и умещается в 100-200 строк
- Все шаги обязательны в любом контексте — проще держать в одном файле
- Логика линейная и не делится на независимые блоки

### Процесс создания подскилла

1. **Определи границы.** Каждый подскилл — это самостоятельный блок: у него своя задача, свои шаги, свой результат. Не дроби на слишком мелкие части.

2. **Запиши в Supabase.** Подскиллы хранятся в таблице `sub_skills`:
   ```sql
   INSERT INTO sub_skills (skill_id, name, description, instruction, sort_order, status)
   VALUES (
     '<id_главного_скилла>',
     'название-подскилла',       -- kebab-case
     'Краткое описание',          -- 1-2 предложения
     '## Полная инструкция...', -- Markdown
     1,                          -- порядок выполнения
     'active'
   );
   ```

3. **Добавь секцию `## Подскиллы` в главный скилл.** В поле `instruction` главного скилла добавь таблицу:
   ```markdown
   ## Подскиллы

   При активации этого скилла автоматически загружай следующие подскиллы:

   | Подскилл | Описание | Когда применять |
   |----------|----------|-----------------|
   | `css-animations` | Анимации и переходы | Всегда |
   | `ssr-setup` | Настройка серверного рендеринга | Только если проект SSR |
   ```

4. **Проверь порядок.** `sort_order` определяет последовательность загрузки. Если подскилл B зависит от результата подскилла A — A должен быть раньше.

### Структура инструкции подскилла

```markdown
## Название подскилла

Краткое описание — что делает этот подскилл.

### Контекст

Этот подскилл является частью скилла `<имя-главного-скилла>`.
Применяется когда: <условие>.

### Процедура

1. Шаг 1
2. Шаг 2
3. ...

### Пример

Input: ...
Output: ...
```

### Принципы подскиллов

- **Не дублируй.** Общие вещи (зависимости, API-ключи, форматы) — в главном скилле. Подскилл наследует контекст.
- **Один уровень вложенности.** Подскилл НЕ может иметь своих подскиллов. Максимум: скилл → подскиллы.
- **Опциональность через колонку «Когда применять».** Не все подскиллы нужны всегда — указывай условие.
- **Автономность.** Подскилл должен быть понятен без чтения других подскиллов (но может ссылаться на контекст главного скилла).
